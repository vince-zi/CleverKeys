# Release Process Skill

Use this skill when tagging, releasing, or preparing a new version of CleverKeys for GitHub and F-Droid distribution.

**Trigger words** (any of these in the user's message → load this skill FIRST):
- "release", "publish", "tag", "version bump"
- "F-Droid", "fdroid"
- "changelog", "fastlane"
- "dry run release", "prepare release"

## Two-Channel Changelog Model (CRITICAL)

CleverKeys ships **two separate changelog channels** plus a third human-readable file. Each new release MUST update all three:

| Channel | Location | Audience | Format |
|---|---|---|---|
| **GitHub release body** | Generated by `release.yml` | Obtainium, sideloaders | Markdown, ~30-60 lines |
| **F-Droid in-app changelog** | `fastlane/metadata/android/en-US/changelogs/{baseCode}{abi}.txt` × 3 | F-Droid app users | Plaintext, ≤30 lines |
| **Project history** | `CHANGELOG.md` (Keep-A-Changelog format) | Repo readers | Markdown, structured headings |

**Common pitfall**: `metadata/fdroid/tribixbite.cleverkeys.yml` is the **F-Droid build recipe** (commits, ABI splits, prebuild hooks) — NOT a changelog. Don't put release notes there. Its `CurrentVersion:` field is auto-updated by F-Droid's bot post-publish, not manually.

**Common pitfall**: `CHANGELOG.md` may be missing the most recent release entry while fastlane + GitHub bodies are present. The fastlane channel and GitHub release body are the load-bearing changelogs; `CHANGELOG.md` is a secondary record that's been intermittently maintained.

## Pre-Release Checklist

Before starting a release:
- [ ] All planned features/fixes are merged
- [ ] Tests pass: `./gradlew runPureTests` (or skip on Termux — too slow; rely on CI)
- [ ] Build succeeds: `./gradlew compileDebugKotlin`
- [ ] Manual testing completed
- [ ] `memory/todo.md` updated with completed items
- [ ] No `0words.py`-style stray scratch files in `git status`
- [ ] Last `release.yml` run on previous tag was green (`gh run list --workflow=release.yml --limit 1`)
- [ ] F-Droid has built the previous release without error (see "F-Droid API Queries" below)

## Version Code Scheme

### Base Version Code
```
versionCode = VERSION_MAJOR * 10000 + VERSION_MINOR * 100 + VERSION_PATCH
```

Examples:
| Version | Base versionCode |
|---------|------------------|
| v1.2.5  | 10205           |
| v1.2.6  | 10206           |
| v1.3.0  | 10300           |
| v2.0.0  | 20000           |

### ABI-Specific Version Codes
```
abiVersionCode = baseVersionCode * 10 + abiSuffix
```

| ABI         | Suffix | v1.2.5 Code | v1.2.6 Code |
|-------------|--------|-------------|-------------|
| armeabi-v7a | 1      | 102051      | 102061      |
| arm64-v8a   | 2      | 102052      | 102062      |
| x86_64      | 3      | 102053      | 102063      |

## Dry-Run Release Procedure (when user says "dry run prepare release")

If the user requests a dry run, deliver this exact structure WITHOUT modifying files or pushing tags:

1. **Current state snapshot**:
   - Latest tag (`git describe --tags --abbrev=0`)
   - Current `build.gradle` VERSION_* values
   - Untracked files (`git status --short`)
   - Substantive commits since last tag (`git log <tag>..HEAD --pretty=format:"%h %s"` filtered to feat/fix/perf/refactor)

2. **F-Droid current published state** (use `f-droid.org/api/v1/packages/tribixbite.cleverkeys`)

3. **Recommendation**: MAJOR / MINOR / PATCH with rationale based on **change content**, not commit count
   - PATCH: bug fixes, perf, polish on shipped features
   - MINOR: new user-facing features, schema additions, new tabs/panels
   - MAJOR: breaking changes, schema-incompatible migrations, full rewrites

4. **Theme groups** (4-8 groups, headline vs supporting). Read enough commit *bodies* (not just titles) to surface the actual user-facing impact. Spawn a subagent for the audit if there are >50 commits — it's faster and avoids context bloat.

5. **Risks/blockers**:
   - Open items in `memory/todo.md`
   - Reproducibility risk audit (`git diff <tag>..HEAD -- build.gradle gradle/wrapper/`)
   - CI status (last `release.yml` run, recent `ci.yml` failures)

6. **Drafted changelogs** for all three channels (GitHub body, fastlane, CHANGELOG.md)

7. **File list** to edit before tagging (with line numbers)

8. **Suggested command sequence** but do NOT execute it.

## Comprehensive Audit Pattern (for >50 commits since last tag)

Spawn a `general-purpose` subagent with this prompt template:

```
Audit CleverKeys at <repo path> from <last tag>..HEAD for release prep.

1. Run `git log <tag>..HEAD --pretty=format:"%h %s%n%b%n---"` and read ALL substantive commits (feat/fix/perf/refactor — skip docs/test/chore).
2. For cryptic titles, run `git show --stat <hash>` to see touched files.
3. Group into 4-8 themes (headline vs supporting). User-facing language only — no commit-hash refs, no "refactor".
4. Read previous release's fastlane changelogs (last 3 versions in `fastlane/metadata/android/en-US/changelogs/`) for tone/length match.
5. Read `gh release view <last-tag>` for GitHub body style.
6. Check `memory/todo.md` for incomplete/blocked items.
7. List excluded commits (test-only, doc-only, CI-only) with reason.

Deliver: recommendation (MAJOR/MINOR/PATCH + rationale), themes with bullets, blockers, three drafted changelogs (GitHub markdown, fastlane plaintext, CHANGELOG.md Keep-A-Changelog).
```

Run a parallel sanity-check subagent for F-Droid status + reproducibility:

```
Release-readiness sanity check for CleverKeys at <repo path>.

1. F-Droid API queries (see release-process.md "F-Droid API Queries" section)
2. Compare local `metadata/fdroid/tribixbite.cleverkeys.yml` to upstream at `gitlab.com/fdroid/fdroiddata/-/raw/master/metadata/tribixbite.cleverkeys.yml`
3. `git diff <tag>..HEAD -- build.gradle gradle.properties gradle/wrapper/` — flag any AGP/Gradle/Kotlin/dependency bumps (LOW/MEDIUM/HIGH reproducibility risk)
4. CI status: last release.yml + recent ci.yml runs
5. Untracked files verdict (each file: keep / .gitignore / delete)
6. Test status (or skip if Termux)

Deliver: structured report with each section as a heading.
```

## Step-by-Step Release Process

### 1. Query Current State

```bash
# Current F-Droid version
curl -s "https://f-droid.org/api/v1/packages/tribixbite.cleverkeys" | jq -r '"F-Droid: v" + .suggestedVersionName + " (code " + (.suggestedVersionCode|tostring) + ")"'

# Current GitHub release
gh release list --limit 1

# Current git tag
git describe --tags --abbrev=0

# Current build.gradle version
grep "VERSION_" build.gradle | head -3
```

### 2. Determine New Version

Semantic versioning:
- **MAJOR**: Breaking changes, major rewrites
- **MINOR**: New features, backwards-compatible
- **PATCH**: Bug fixes, minor improvements

### 3. Update build.gradle

Edit `build.gradle` lines 58-60:
```gradle
ext.VERSION_MAJOR = 1
ext.VERSION_MINOR = 2
ext.VERSION_PATCH = 6   // <-- increment this
```

### 4. Write Fastlane Changelogs

Create THREE identical changelog files:

```bash
# Calculate version codes
NEW_VERSION="1.2.6"
BASE_CODE=10206  # MAJOR*10000 + MINOR*100 + PATCH

# File paths
CHANGELOG_DIR="fastlane/metadata/android/en-US/changelogs"
touch "$CHANGELOG_DIR/${BASE_CODE}1.txt"  # armv7
touch "$CHANGELOG_DIR/${BASE_CODE}2.txt"  # arm64
touch "$CHANGELOG_DIR/${BASE_CODE}3.txt"  # x86_64
```

### Changelog Format

```
vX.Y.Z - Short Title

LAYOUT NOTE: (optional - only for layout changes)
Description of layout changes if any.

NEW FEATURES:
• Feature 1 - Brief description
  - Detail line if needed

• Feature 2 - Brief description

FIXES:
• Fixed issue description (issue #XX)
• Fixed another issue

IMPROVEMENTS:
• Performance improvement description
• Code cleanup description
```

### Example Changelog

```
v1.2.6 - Emoji Panel Polish

NEW FEATURES:
• Emoji long-press tooltip - Shows emoji name above pressed cell
  - Works with all emoji including flags and emoticons
  - Auto-dismisses after 2.5 seconds

• Emoticons category - 119 text emoticons in emoji picker
  - Searchable via keywords (shrug, lenny, tableflip)
  - Length-based scaling for proper display

FIXES:
• Emoji/clipboard panel gap eliminated
• App-switch no longer causes empty panels
• Flag emoji names now show country (Japan vs Regional Indicator)
```

### 5. Commit Version Bump

```bash
# Stage changes
git add build.gradle fastlane/

# Commit with version in message
git commit -m "chore: bump version to v1.2.6

- Update VERSION_PATCH in build.gradle
- Add fastlane changelog for v1.2.6

-- claude-opus-4-5"
```

### 6. Create and Push Tag

```bash
# Create annotated tag
git tag -a v1.2.6 -m "v1.2.6 - Emoji Panel Polish"

# Push commit and tag
git push origin main
git push origin v1.2.6
```

### 7. Verify GitHub Release

GitHub Actions (`release.yml`) automatically:
1. Verifies tag matches build.gradle version
2. Builds per-ABI APKs (matching F-Droid exactly)
3. Signs with release keystore
4. Creates GitHub Release with changelog
5. Uploads APK artifacts

Check progress:
```bash
gh run list --workflow=release.yml --limit 1
gh run view <run-id>
```

### 8. Monitor F-Droid Build

F-Droid typically picks up new tags within 24-48 hours:

```bash
# Check if build started
curl -s "https://f-droid.org/repo/status/running.json" | jq '.running[] | select(.appid == "tribixbite.cleverkeys")'

# Check build result (after completion)
curl -s "https://f-droid.org/repo/status/build.json" | jq '.[] | select(.appid == "tribixbite.cleverkeys")'
```

## Changelog Generation Script

For generating changelog from commits since last tag:

```bash
#!/bin/bash
LAST_TAG=$(git describe --tags --abbrev=0 HEAD^)
NEW_TAG=$1

echo "v${NEW_TAG#v} - Title Here"
echo ""
echo "Changes since $LAST_TAG:"
echo ""

# Group commits by type
git log $LAST_TAG..HEAD --pretty=format:"• %s" | while read line; do
  if echo "$line" | grep -qi "feat\|add\|new"; then
    echo "NEW: $line"
  elif echo "$line" | grep -qi "fix"; then
    echo "FIX: $line"
  else
    echo "OTHER: $line"
  fi
done
```

## Lessons Learned

### Don't conflate the build recipe with the changelog
- `metadata/fdroid/tribixbite.cleverkeys.yml` is the F-Droid build recipe — versionCode, commit, ABI splits, prebuild hooks. Don't put release notes here. Its `CurrentVersion:` field is auto-updated by F-Droid's bot post-publish, not manually.
- Release notes for F-Droid go in `fastlane/metadata/android/en-US/changelogs/{baseCode}{abi}.txt` — **THREE separate files per release**, one per ABI.

### Don't undercount work
- 100+ commits with mostly `fix:` titles can still be a MINOR release if those fixes ship a new tab, new schema, or new modal. Read commit *bodies* and *file diffs*, not just titles. If in doubt, spawn an audit subagent — it's faster and avoids context bloat.
- Group themes BY user-facing surface (clipboard, dictionary, settings panel, web demo), not by commit prefix (`feat` vs `fix`).

### Two concurrent changelog channels
- F-Droid users see `fastlane/.../changelogs/{code}.txt` inside the F-Droid app. Plaintext, ≤30 lines, • bullets.
- GitHub release page users see the body generated by `release.yml`. Markdown, longer, headers and bullets.
- Both must exist for a release. The fastlane file is what's missing 99% of the time; people remember CHANGELOG.md and forget the per-ABI fastlane files.

### Reproducibility risk audit before tagging
- Always run `git diff <last-tag>..HEAD -- build.gradle gradle.properties gradle/wrapper/`. Any AGP/Gradle/Kotlin version bump = HIGH risk; F-Droid's verifier will reject byte-mismatched builds.
- Adding `androidTestImplementation` deps doesn't affect the release APK (test-scoped). Adding `implementation` deps does.

### Untracked files at release time
- Stray scratch files (`0words.py`, ad-hoc word lists) are NOT in the tagged tree because `git archive` only includes tracked files. So they don't leak into the release. But they CAN signal incomplete in-progress work — pause and ask before tagging.

### Fastlane changelog 500-char limit (soft, but aim for it)
- F-Droid docs specify `max 500 chars` for `changelogs/{versionCode}.txt`. The limit is a fastlane Play Store inheritance — F-Droid does NOT reject overlong files but truncates the in-app "What's New" preview.
- **Project precedent matters**: prior CleverKeys releases ranged from 500 bytes (v1.3.0, engineered to the cap) to 2581 bytes (v1.2.8, way over). The 500-byte ceiling is the target — match v1.3.0's discipline.
- **Char vs byte ambiguity**: F-Droid's spec says "chars" but the `•` bullet is 3 bytes / 1 char in UTF-8. v1.3.0 fit BOTH interpretations: 500 bytes / 482 chars. Aim for ≤500 bytes (stricter) to satisfy either reading.
- **Verify with `wc -c -m <file>`** before saving. The `-c` is bytes, `-m` is chars. Both must be ≤500.
- **Quickest trim**: lean on issue numbers — "Scoped-storage import (#70)" reads the same as "Scoped-storage import fallbacks (#70)" once the user clicks the issue link, but saves 10 chars.
- **What to cut first if over budget**: WEB section (web demo isn't part of the F-Droid app), then PERF detail (combine into one line), then trailing per-feature elaborations. Headlines are non-negotiable.

## Troubleshooting

### Tag/Version Mismatch
If GitHub Actions fails with "VERSION MISMATCH":
1. Ensure build.gradle VERSION_MAJOR/MINOR/PATCH matches tag
2. Delete the tag: `git tag -d vX.Y.Z && git push origin :refs/tags/vX.Y.Z`
3. Fix build.gradle, commit, re-tag

### F-Droid Not Picking Up Release
1. Check metadata file exists in fdroiddata repo
2. Verify tag is annotated (not lightweight)
3. Check F-Droid build logs at https://monitor.f-droid.org/

### Reproducibility Issues
F-Droid requires byte-identical builds. If verification fails:
1. Check `build.gradle` reproducibility settings
2. Verify Java 17, SDK 34, build-tools 34.0.0
3. See `.github/workflows/release.yml` for exact build steps

## Release Cadence

- **Major releases** (X.0.0): Major rewrites, breaking changes
- **Minor releases** (X.Y.0): New features, significant improvements
- **Patch releases** (X.Y.Z): Bug fixes, polish, small improvements

Typical timeline:
- Development: variable
- GitHub Release: Immediate after tag push (~5 min build)
- F-Droid: 24-48 hours after tag (automated detection)
